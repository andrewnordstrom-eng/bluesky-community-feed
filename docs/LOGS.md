# Log and Journal Types

This repo keeps several distinct written records. They are not interchangeable.
Before writing to one, confirm it is the right record for what you are noting.
This file is the index of record; when in doubt, start here.

## Quick decision

- Measured experiment or research finding you are still iterating on (metrics, evals,
  simulations) — `docs/eval-lab-journal.md` (local-only).
- A reviewed evidence packet that backs a specific claim or PR, with reproducible commands
  and checksums — `docs/lab/YYYY-MM-DD-<topic>.md` plus the `artifacts/lab/` protocol.
- What shipped, chronologically, per PR or phase — `docs/dev-journal.md`.
- Proof that an issue's closeout gates passed — `ops/receipts/`.
- Live service behaviour on the VPS (crashes, disk, retention) — `journalctl`.

## The records

| Record | Tracked in git | Purpose | Append rule |
|--------|----------------|---------|-------------|
| `docs/dev-journal.md` | Yes | Chronological "what shipped" development log for the whole repo, per PR or phase. Link to a `docs/lab/` packet for numbers rather than pasting metrics here. | New dated entry at the bottom; never prepend or edit past entries. |
| `docs/lab/YYYY-MM-DD-<topic>.md` | Yes | Dated, point-in-time evidence packets tied to a Linear issue (production metrics pulls, ingestion/voting validation runs, audits). Narrow in scope and explicit about what they do not prove. Every claim cites a manifest path and a summary path. | One new dated file per audit or run; do not edit past packets. |
| `artifacts/lab/` | Contract only | The reproducibility protocol behind the `docs/lab/` packets. `README.md` and `manifest.schema.json` are committed; per-run payloads under `artifacts/lab/<ISSUE>/<run-id>/` (a `manifest.json` validating against the schema, `checksums.sha256`, one subdirectory per harness phase, and raw receipts) are git-ignored. A claim without a manifest and checksums is not evidence. | One new `<ISSUE>/<run-id>/` directory per run; payloads are regenerated, not edited. |
| `docs/eval-lab-journal.md` | No (local-only) | Running research/findings journal for the topic-classifier evaluation and measured-quality work, with a Measurements section per entry. Excluded from git tracking via `.git/info/exclude`; it will not exist in a fresh clone or CI checkout. Keep the single canonical copy in the main checkout's `docs/` and append there from any worktree. | Append the next numbered entry (`## #NNN — YYYY-MM-DD — Title`); never edit old ones. |
| `ops/receipts/` | Yes | Machine-validated closeout proof, one tree per Linear issue. Generated by the closeout flow and validated in CI. Read these; do not hand-edit them. | Generated by the closeout flow; never hand-edit. |
| `journalctl` (VPS) | Runtime | Live operational logs on the server, not a repo file. Read-only ops, for example `sudo journalctl -u bluesky-feed -f`. See section 9 of `docs/agent/REPO_CONTRACT.md` for the full set of service and alert units. | Not applicable (runtime stream, read-only). |

## Conventions

- Journals are append-only: never edit or delete an old entry; always append a new one with
  the next sequence number or date.
- `docs/dev-journal.md`, `docs/lab/`, and the `artifacts/lab/` contract are committed history
  and ship with the repo; write them in the repo's normal, tool-neutral voice.
- `docs/eval-lab-journal.md` is a local working log only. It is never pushed. Anything from it
  that graduates into a committed artifact (a `docs/lab/` packet, the dev journal, the paper)
  must be rewritten in that same committed voice.

## Not sure which one?

- Backs a claim you will ship or cite — `docs/lab/` plus `artifacts/lab/`.
- Ongoing research or measurements still in flight — `docs/eval-lab-journal.md`.
- Just recording that work merged — `docs/dev-journal.md`.
